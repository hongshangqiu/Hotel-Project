# 参考vue-element-admin的建议
## 现有问题
列表页重复请求：首次渲染时同时触发 useEffect([]) 和 useEffect([page])，会重复加载/分页错乱
类型定义重复：UserRole/IUser 在两个文件里重复定义，易导致类型分裂与误导引用
全局请求封装未使用且跨端风险：axios 在小程序端默认不可用，且项目内没有任何引用（如果我们不考虑展示小程序端可忽略）
其他：登录页检测 useEffect 无依赖、且 isLoginPage 状态未使用，造成不必要渲染。搜索默认日期硬编码为固定日期，时间推移后会导致默认时间过期
## 结构
vue-element-admin 有明确的“权限/路由/请求/Mock/全局状态”分层与规范化目录；我们的项目目前分层更轻，部分功能散落在页面中（例如登录校验、路由跳转、权限判断）。
vue-element-admin 的请求层与权限控制清晰；我们的请求封装未使用，权限/路由更多靠页面内部逻辑与 app.tsx 处理，后期维护成本较高。
vue-element-admin Mock 与数据层可替换；我们的项目依赖 LocalStorage 的 Mock，跨端数据一致性需额外处理。
## 可改进
统一跨端导航与外链：在 Taro 页面里直接用 window 导致小程序端报错/白屏，考虑全部改成 Taro 官方 API
简化列表请求逻辑：统一由一个 useEffect 驱动分页请求，避免重复加载。
合并类型来源：保留一个类型文件作为唯一出口，避免 UserRole 多处定义。
正式落地 request 封装并替换本地 Mock。
